## Stage 3: Virtual Filesystem with IDBFS

*Corresponds to feasibility doc Phase 3 + Section 11.*

### 3.1 — Emscripten MEMFS + IDBFS setup

- [ ] Configure Emscripten FS in the WASM module init:
  - Mount IDBFS at `/home` for persistent user files
  - Mount MEMFS at `/tmp` for scratch space
  - `/library` is read-only (embedded at build time via `--embed-file`)
- [ ] Call `FS.syncfs(true, ...)` on startup to populate MEMFS from IndexedDB
- [ ] Expose `etil_sync_fs()` C function for JS to trigger sync after writes

### 3.2 — Wire ETIL file words to Emscripten FS

- [ ] Re-enable file I/O primitives for WASM build (currently excluded)
  - Use Emscripten's POSIX-compatible `fopen`/`fread`/`fwrite` (backed by MEMFS)
  - These are synchronous in WASM — no libuv needed
- [ ] Alternative: keep file I/O excluded, implement via JS interop:
  - `etil_write_file(path, data)` → `FS.writeFile()` + `FS.syncfs()`
  - `etil_read_file(path)` → `FS.readFile()`
  - Register as TIL primitives via the WASM entry point
- [ ] Decision needed: which approach is simpler. Emscripten's POSIX FS may "just work" since the C++ file I/O code uses standard `<fstream>` in the sync fallback path.
- [ ] Test: `write-file`, `read-file`, `ls`, `cat`, `include` over MEMFS

### 3.3 — Automatic sync to IndexedDB

- [ ] TypeScript glue: `FS.syncfs(false, callback)` after every file-mutating operation
  - Hook into `etil_interpret()` return — if the line contained a file-writing word, sync
  - Simpler: sync on a 30-second heartbeat timer + `beforeunload`
- [ ] Handle sync errors gracefully (quota exceeded → status bar warning)

### 3.4 — Pre-populated example files

- [ ] Bundle example `.til` files into `/home/examples/` via `--embed-file` or write them on first launch
- [ ] First-launch detection: if IDBFS `/home` is empty after sync, copy examples from embedded FS
- [ ] Examples: `hello.til`, `fibonacci.til`, `factorial.til`, `observable-demo.til`

### 3.5 — File upload (drag-and-drop + /upload)

- [ ] TypeScript: listen for `dragover`/`drop` events on terminal container
- [ ] Read dropped files via `FileReader` API
- [ ] Write to MEMFS at `/home/<filename>` via `FS.writeFile()`
- [ ] Sync to IDBFS
- [ ] Display confirmation: "Uploaded <filename> (N bytes)"
- [ ] `/upload` meta-command: open file picker dialog as alternative to drag-and-drop

### 3.6 — File download (/download)

- [ ] `/download <path>` meta-command
- [ ] Read file from MEMFS via `FS.readFile()`
- [ ] Create `Blob` + `URL.createObjectURL()` + trigger download via hidden `<a>` element
- [ ] Clean up object URL after download

### 3.7 — Bulk export/import (/export, /import)

- [ ] `/export` meta-command:
  - Walk MEMFS `/home` recursively via `FS.readdir()` + `FS.readFile()`
  - Build tar archive (512-byte headers + file data — implement minimal tar writer, no library needed)
  - Compress via `CompressionStream('gzip')` (native browser API, no pako.js)
  - Trigger download: `etil-home-YYYYMMDD-HHMMSS.tar.gz`
- [ ] `/import` meta-command:
  - Open file picker for `.tar.gz`
  - Decompress via `DecompressionStream('gzip')`
  - Parse tar headers, extract files
  - Write to MEMFS via `FS.writeFile()`, create dirs via `FS.mkdir()`
  - Sync to IDBFS
  - Size guard: reject archives > 50 MB decompressed
  - Display: "Imported N files into /home"
- [ ] Drag-and-drop `.tar.gz` files triggers import automatically

## Stage 4: HTTP via fetch()

*Corresponds to feasibility doc Phase 4.*

### 4.1 — JavaScript fetch() wrapper

- [ ] Create `web/src/fetch-bridge.ts`:
  - `async fetchGet(url: string, headers: Record<string,string>): Promise<{body: Uint8Array, status: number}>`
  - `async fetchPost(url: string, headers: Record<string,string>, body: Uint8Array): Promise<{body: Uint8Array, status: number}>`
  - Handle CORS errors → return descriptive error, not browser exception
  - Handle network errors → return error with status 0

### 4.2 — Wire fetch to ETIL interpreter

- [ ] Two approaches (decide during implementation):
  - **Option A:** Register `http-get`/`http-post` as WASM-specific C primitives that call into JS via `EM_ASM` or `emscripten_fetch()`
  - **Option B:** Handle `http-get`/`http-post` in the TypeScript glue — intercept these words before passing to WASM, execute fetch, push results back
- [ ] Option B is simpler but breaks the "interpreter handles everything" model
- [ ] Option A requires `-sASYNCIFY` (fetch is async) — same issue as temporal observables
- [ ] **Pragmatic choice:** Option B — glue intercepts `http-get`/`http-post` as meta-commands: `/fetch GET <url>`, or detect them in the interpret output and handle in JS

### 4.3 — CORS user experience

- [ ] When fetch fails due to CORS, display clear error:
  ```
  Error: CORS blocked request to <url>
  The target server must set Access-Control-Allow-Origin headers.
  ```
- [ ] Do not expose raw browser error messages
- [ ] `/fetch` help text explains CORS limitations

### 4.4 — URL validation

- [ ] Reuse ETIL's URL validation (already compiled into WASM)
- [ ] SSRF protection less critical in browser (browser enforces same-origin)
- [ ] Keep domain allowlist check as defense-in-depth

## Stage 5: Testing & Optimization

*Corresponds to feasibility doc Phase 5.*

### 5.1 — Automated test suite for WASM

- [ ] Create `tests/wasm/` directory with Node.js test runner
- [ ] Port subset of native tests to run against WASM module in Node:
  - Arithmetic: `42 21 + .` → `63`
  - Stack ops: `1 2 3 rot .s` → `(3) 2 3 1`
  - Colon defs: `: double dup + ; 21 double .` → `42`
  - Control flow: `: fact dup 1 > if dup 1 - recurse * then ; 10 fact .`
  - Strings: `s" hello" s" world" s+ .`
  - Arrays: `array-new 1 array-push 2 array-push array-length .`
  - JSON: `j| {"a":1} | s" a" json-get .`
  - Observables: `0 10 obs-range obs-to-array array-length .`
  - Selection/Evolution: `select-strategy`, `evolve-status`
- [ ] Run tests via `pnpm test` script
- [ ] Verify temporal ops return "Unknown word" (not crash)
- [ ] Verify matrix ops return "Unknown word" (not crash)
- [ ] Verify file I/O ops behavior (error or MEMFS depending on Stage 3)
- [ ] Verify `sleep` returns error message

### 5.2 — Binary size optimization

- [ ] Run `wasm-opt -Oz` on output (already in build)
- [ ] Measure uncompressed + gzipped sizes
- [ ] If > 2 MB gzipped: analyze with `twiggy top` or `wasm-objdump --section`
- [ ] Consider: `-s MINIMAL_RUNTIME=1` (drops unused Emscripten features)
- [ ] Consider: `-Os` instead of `-O2` for the C++ compilation
- [ ] Consider: `--closure 1` for Emscripten JS minification
- [ ] Target: < 500 KB gzipped (currently 336 KB — already excellent)

### 5.3 — Performance benchmarks

- [ ] Create `benchmarks/wasm/` with comparative tests:
  - `10000 0 do i drop loop` — loop overhead
  - `: fib dup 1 > if dup 1 - recurse swap 2 - recurse + then ; 30 fib` — recursion
  - String concatenation: 1000x `s+`
  - Array operations: 10000 pushes + pops
  - Observable pipeline: `0 10000 obs-range obs-map obs-to-array`
- [ ] Compare WASM vs native REPL times
- [ ] Report in design doc

### 5.4 — Cross-browser testing

- [ ] Chrome (primary — currently tested)
- [ ] Firefox
- [ ] Safari (if available — may need macOS)
- [ ] Edge (Chromium — should match Chrome)
- [ ] Mobile Chrome (Android) — does the REPL work on mobile?
- [ ] Document any browser-specific issues

### 5.5 — Error handling hardening

- [ ] Infinite loop protection: ensure `ctx.tick()` budget works in WASM
  - Test: `: hang begin again ; hang` — should error, not freeze tab
- [ ] Stack overflow protection: deep recursion should error, not crash WASM
- [ ] Memory exhaustion: large string/array allocations should error gracefully
- [ ] Test all error paths produce user-visible messages, not silent failures
